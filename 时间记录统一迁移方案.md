# 时间记录统一迁移方案

## 1. 目标与边界

### 1.1 目标

把当前拆分为 `time_records` 与 `learning_records` 的“时间类事实”统一为一条主数据链：

- 顶层实体统一为 `time_record`
- `学习 / 工作 / 娱乐 / 生活 / 休息 / 社交` 统一作为 `time_record.category_code`
- “具体做了什么”统一沉淀到 `time_record.content`
- 学习专属语义不再通过独立表表达，而是作为 `time_record` 的可选扩展字段表达

### 1.2 这次迁移不是 MVP

本次不做“仅改 UI 的兼容层”或“长期双写双读”的过渡方案。迁移目标是模块化重构，但落点是单一主模型，而不是保留两套长期并存的数据链。

### 1.3 迁移边界

本次覆盖：

- 数据库 schema 与历史数据迁移
- Rust model / repository / service / FFI
- AI 解析 prompt、draft schema、commit 规则
- Flutter 录入、审核、列表、编辑、项目详情、统计展示
- 导出、demo、测试

本次不覆盖：

- 收入/支出模型重构
- 项目、标签、自定义维度的领域重构
- review note / ignored context 的语义改造

## 5. 迁移目标模型

## 5.1 不直接沿用当前 `time_records`

不能直接把 `learning_records` 简单并入现有 `time_records`，原因有两个：

1. 现有 `time_records` 没有 `content`
2. 现有 `time_records` 强制要求 `started_at` 和 `ended_at`

而当前 `learning_records` 明确允许：

- 只有 `occurred_on + duration_minutes`
- 没有完整时间窗

如果目标模型仍强制完整时间窗，就无法无损吸收历史学习数据。

## 5.2 目标是“重构后的 time_records”

建议把 `time_records` 重构为统一时间事实表，字段语义如下：

- `id`
- `user_id`
- `occurred_on`
  - 本地日期锚点
  - 对 duration-only 记录是必需字段
- `started_at` nullable
- `ended_at` nullable
- `duration_minutes` not null
- `category_code` not null
- `content` not null
- `application_level_code` nullable
  - 仅 `category_code='learning'` 时有业务意义
- `efficiency_score` nullable
- `value_score` nullable
- `state_score` nullable
- `ai_assist_ratio` nullable
- `note` nullable
- `source`
- `parse_confidence`
- `is_public_pool`
- `is_deleted`
- `created_at`
- `updated_at`

建议增加一致性约束：

- `started_at` 与 `ended_at` 要么同时为空，要么同时非空
- 若两者非空，则：
  - `ended_at > started_at`
  - `duration_minutes` 必须与时间窗一致或允许极小误差
- `occurred_on` 必须始终存在
- `content` 不能为空
- `application_level_code` 仅在 `category_code='learning'` 时使用

## 5.3 类别与内容的层级

统一后语义应为：

- `category_code = learning`
- `content = 写作业`

而不是：

- 记录类型 = learning
- 标题 = 写作业

即：

- 类别是分类维度
- 内容是行为主体

## 9. 数据迁移设计

## 9.1 迁移前提

由于当前 `time_records` 不支持 duration-only，迁移必须先扩 schema，再迁历史数据。

不能先把 `learning_records` 硬塞到旧 `time_records`。

## 9.2 新旧字段映射

### A. `time_records(category != learning)` -> 新 `time_records`

直接迁移，主要补充：

- `occurred_on = 本地 started_at 对应日期`
- `content`
  - 默认从 `note` 提炼
  - 若无法可靠提炼，可先回填为空占位值，但最终必须补齐

### B. `time_records(category = learning)` -> 新 `time_records`

映射：

- `category_code = learning`
- `content`
  - 由旧 `note` 提炼
  - 若 `note` 为空，回填默认内容例如“学习”
- `application_level_code = null`
- 其余时间与评分字段直接迁移

### C. `learning_records` -> 新 `time_records`

映射：

- `occurred_on -> occurred_on`
- `started_at -> started_at`
- `ended_at -> ended_at`
- `duration_minutes -> duration_minutes`
- `content -> content`
- `application_level_code -> application_level_code`
- `efficiency_score -> efficiency_score`
- `ai_assist_ratio -> ai_assist_ratio`
- `note -> note`
- `value_score = null`
- `state_score = null`
- `category_code = learning`

## 9.3 关联表迁移

`record_project_links`：

- `record_kind='learning'` 全部改写为 `record_kind='time'`
- `record_id` 指向新时间记录 id

`record_tag_links`：

- `record_kind='learning'` 全部改写为 `record_kind='time'`
- `record_id` 指向新时间记录 id

## 9.4 ID 迁移策略

建议：

- 为 `learning_records -> time_records` 生成新 id
- 保留一张临时映射表：
  - `legacy_learning_record_id`
  - `new_time_record_id`

原因：

- 可以显式追踪回填关系
- 方便修复 `record_project_links` / `record_tag_links`
- 方便测试和审计

不建议直接复用原 id，除非确认所有调用边界、主键冲突和导出兼容都可控。

## 9.5 历史数据质量问题

迁移前必须识别：

1. `learning_records` 中没有 `started_at/ended_at` 的记录比例
2. `time_records(category=learning)` 中 `note` 为空的比例
3. 同一天同时存在：
   - `time_records(category=learning)`
   - `learning_records`
   的重复记录风险

迁移文档需要把这些作为预检 SQL 单独执行。

## 13. 风险与难点

## 13.1 最大 schema 风险：duration-only 记录

如果目标模型仍要求完整时间窗，这次统一必然失败，因为历史学习记录并不都具备 `started_at/ended_at`。

所以统一必须包含：

- `occurred_on`
- duration-only 支持

## 13.2 最大语义风险：旧 time learning 记录没有明确 content

`time_records(category=learning)` 目前没有 `content`，很多内容混在 `note` 里。迁移时会面临：

- 需要从 `note` 提取内容
- 或接受部分记录只能回填默认内容

这个问题需要迁移前审计。

## 13.3 最大计算风险：项目成本口径变化

统一后学习投入会进入项目时间成本，项目 ROI 会变化。这不是 bug，而是口径修正，但需要：

- 明确告知
- 在验证阶段重点检查

## 13.4 最大接口风险：前后端删除 learning 分支后会有广泛编译影响

这属于预期影响，不应通过“保留空壳接口”回避。应一次性完成模块清理。

---

## 14. 建议的实施原则

1. 先统一 schema，再统一 AI，再统一前端，不反过来做。
2. 不做长期双写。
3. 不保留 `learning_records` 作为历史兼容表。
4. 所有统计口径必须在同一迭代完成切换。
5. 所有列表标题统一从 `content` 出发，不再让 `category_code` 充当主标题。

---

## 15. 结论

这次要解决的不是“学习页面长得不像时间页面”，而是：

- 当前系统把“学习”错误建模成了与“时间”平级的记录类型
- AI、手动录入、项目成本、日/周复盘、项目详情、快照统计都因此出现双源口径

正确的统一方向是：

- 保留且重构 `time_records`
- 移除 `learning_records`
- 把学习变回 `time_record.category_code='learning'`
- 把“做了什么”统一落在 `time_record.content`
- 把学习专属信息作为时间记录的可选扩展字段

只有这样，数据结构、AI 解析、编辑器、列表、统计、项目成本才会回到同一条主数据链。
