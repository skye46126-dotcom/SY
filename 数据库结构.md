# SkyeOS Rust 数据库完整设计

## 设计目标

这版数据库不再按 MVP 收缩，而是直接按 SkyeOS 的完整核心域来建模。

目标有四个：

- 保留旧项目经过多次演进后验证过的核心结构
- 用 Rust + SQLite 重新固化时间、收入、支出、学习、项目、标签、成本、回顾、备份这些领域
- 让 Flutter 只负责展示和交互，不再把数据库口径散落到页面里
- 为后续 FFI、旧库导入、报表计算、AI 草稿写入预留稳定结构

## 关键设计原则

### 1. 多用户隔离保留

所有核心业务表继续带 `user_id`，这样：

- 单机单用户可以直接用默认用户
- 后面扩展多用户或工作区时不用重做 schema
- 导入旧库时可以明确归属

### 2. 维度表保留，而且直接数据驱动

旧项目 V012 的方向是对的，所以新库继续保留 `dim_*`。

不过这里做了一个适合 Rust/Flutter 的调整：

- 维度表直接以 `code` 做主键
- Flutter 和 Rust 之间传值直接传 `code`
- 写入时如果遇到新 code，可以自动补进维度表

当前已经支持自动补入：

- `dim_time_categories`
- `dim_income_types`
- `dim_expense_categories`
- `dim_learning_levels`
- `dim_project_status`

### 3. 统一关系表从第一天开始就使用

不再回到旧式的：

- `time_log_project`
- `income_project`
- `expense_project`
- `learning_project`
- `project_tag`
- 其他分散标签表

而是统一用：

- `record_project_links`
- `record_tag_links`

这样 Rust 的 repository 和后续报表逻辑会简单很多。

### 4. 不额外拆 `work_records`

重构文档里曾把 `work_records` 写进过第一版建议表，但从旧项目真实实现看，工作相关数据本质上是 `time_log/time_records` 的属性：

- `ai_assist_ratio`
- `efficiency_score`
- `value_score`
- `state_score`

因此这版没有再造一个重复事实表，而是把工作质量字段继续保留在 `time_records` 上。

这是“完整设计”，不是“偷字段”。
因为这样更贴近旧项目真实口径，也更有利于 ROI、回顾、Today 统计统一计算。

## 当前实现的行为特征

### 写入校验

当前 Rust 层已经做了这些基础约束：

- 用户存在校验
- 关联项目存在校验
- 关联标签存在校验
- 时间字段顺序校验
- 日期格式校验
- 百分比和评分范围校验
- 金额非负校验

### 自动补维度

如果写入时遇到新的分类 code，会自动插入对应维度表并标记为：

- `is_system = 0`
- `sort_order = 1000`

这样后续自定义类别不需要先走后台管理页。

### 关系写入同步完成

创建记录时，Rust 会同时写入：

- 事实表
- `record_project_links`
- `record_tag_links`

不会把关联关系留给 Flutter 端自己拼。

## 测试覆盖

当前测试已经验证了三类能力：

1. migration 能完整创建基础表、扩展表和默认用户
2. 时间记录写入后，Today 总览能正确读取时间、收入、支出、学习
3. `project / tag / time / income / expense / learning` 一整套创建和关系写入是通的

## 后续建议

数据库设计已经从 MVP 提升到完整第一版，下一步最值得继续做的是：

1. 补齐查询接口：项目详情、标签明细、回顾窗口、最近记录
2. 实现成本计算查询：结构化成本、必要支出、项目全成本
3. 增加旧 Android SQLite 导入器
4. 接 Flutter FFI，把 Rust 服务层变成真正可调用的核心能力
