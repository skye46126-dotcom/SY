# 原始素材录入规范

这份说明用于降低 AI 解析和规则兜底的审核成本。系统允许输入很乱的日记、流水账、复盘和计划，但越接近下面的结构，越容易被直接转成可审核草稿。

当前链路原则：

1. AI 先尽量把事件、复盘、忽略上下文分开。
2. UI 只提示用户补缺少的字段，不要求重写全文。
3. 规则层只负责校验、映射、拦脏数据，不重新猜整段语义。

## 一、推荐结构

每天一篇时，建议使用固定标题和分区：

```text
## 9.17 记录

### 时间
06:30 起床
06:40-07:30 背单词 260 个
09:00-11:40 高数课
写公众号 52 分钟

### 支出
早餐 8 元
午餐 18 元

### 学习/产出
背单词 50 分钟，260 个单词
写公众号 52 分钟，完成一篇

### 反思/上下文
今天课程和自我探索之间的平衡需要重新安排。

### 计划
明天早上看健身房是否开门。
```

不要求每次都完整填写这些分区。高频录入可以只写一两行，长期配置和管理类内容最好单独成段。

## 二、系统如何理解不同文本

| 文本形态 | 系统类型 | 是否可直接提交 | 说明 |
| --- | --- | --- | --- |
| `09:00-11:40 上高数` | time_record | 可审核提交 | 完整时间段，规则可计算时长 |
| `背单词 50 分钟` | time_record 或 learning_record | 需要审核 | 有明确时长，但没有时间窗口 |
| `早餐 8 元` | expense_record | 可审核提交 | 有金额，可绑定项目和类别 |
| `工资到账 3000 元` | income_record | 可审核提交 | 有收入金额和来源 |
| `8:20 出寝室买早餐` | time_marker | 不直接提交 | 只有单点时间，是时间线锚点，不是时间消耗记录 |
| `写到 1:32` | time_marker | 不直接提交 | 只有结束点，缺少开始时间或时长 |
| `感觉高数课很水` | reference_note | 不直接提交 | 反思和评价会保存为引用上下文 |
| `英语学习一小时失败` | review_note | 不直接提交 | 失败项、没做成、忘了做，优先作为复盘素材 |
| `任务不明确，效率降低` | review_note | 不直接提交 | 短反思、原因分析、改进点，优先进入复盘 |
| `每月会员 28 元` | recurring_expense_rule | 需要审核 | 长期订阅或固定成本，和日支出分开 |
| `新增项目：公众号` | project | 需要审核 | 管理类变更，不和日常记录混在一起入库 |

## 三、时间录入规则

最容易被正确处理：

```text
09:00-11:40 上课
14:20-15:20 睡觉
背单词 50 分钟
写公众号 52min
```

需要用户审核：

```text
8:20 左右出寝室
11点40下课
写到 1:32
中间一段时间不记得做了什么
```

不建议直接作为时间记录：

```text
上午忙了一会儿
下午一直在弄事情
晚上状态一般
```

如果只有单点时间，系统会保留为 `time_marker`，用于还原一天的时间线，但不会直接算入时间分配。

最推荐的时间写法：

```text
09:00-11:40 上课
5-9点 做 UI 优化
背单词 50 分钟
预习高数课 2h
```

时间锚点最好和动作写在一起，或者紧挨着下一行动作：

```text
下午
5-9点
做 UI 优化
```

## 四、金钱录入规则

支出和收入必须尽量包含金额。建议写法：

```text
早餐 8 元
日餐 84 元，餐饮
买书 36 元，学习
工资到账 3000 元
报销 120 元
```

长期消费和当天支出要分开：

```text
今天午饭 18 元
每月腾讯云服务器 58 元
年度会员 198 元，到期 2027-04-01
```

系统会把当天消费作为 `expense_record`，把长期订阅、月支出基线、资本性支出作为配置草稿处理。

## 五、项目、标签、类别绑定

可以自然写，也可以显式写。显式写法更稳定：

```text
写公众号 52 分钟，项目：公众号，标签：输出
午餐 18 元，项目：校园生活，类别：餐饮
背单词 50 分钟，项目：英语，标签：学习
```

规则层只做名字匹配和候选绑定，不会生成数据库 ID。项目名或标签名不存在时，会进入审核，由用户确认是否新建。

## 六、效率和状态字段

效率、价值、状态、AI 协助比例建议显式给出，不要依赖系统猜测：

```text
写公众号 52 分钟，效率 4/5，价值 5/5，AI 协助 30%
线下活动方案 80 分钟，状态 3/5，风险：没有备用方案
```

如果没有显式分数，系统最多保留为备注或 reference_note，不会硬算。

## 七、复盘和失败项写法

这类内容建议单独一行，系统会更稳定分到复盘素材：

```text
英语学习一小时失败
没有制定好任务量，做完就停了

任务不明确，后面在小鹅通乱看

GPT辅助学习很舒服，效率能提升
```

建议：

- 失败项和原因分析单独写，不要夹在时间段中间。
- 如果某句话既有事件事实，又有感受，可以写成“事件 + 备注”。
- 短反思不要怕碎，系统现在会优先把它们放进复盘，而不是当垃圾文本丢掉。

## 八、适合处理的原始素材

适合：

- 一行一个动作、消费、学习或想法。
- 有明确日期，或者全文共享一个日期标题。
- 时间段、时长、金额尽量写完整。
- 反思、计划、社交、健康状态可以保留，但不要期待它们自动成为时间记录。
- 长期配置单独写，例如订阅、月预算、项目、标签、维度。

不适合直接提交：

- 多天内容混在一段里，没有日期边界。
- 大段连续散文，没有换行、标点或段落。
- 使用大量代词，例如“那个”“这个”“他那件事”，但没有上下文。
- 只写“忙了一下午”“花了不少钱”“学了一些东西”。
- 把当天支出、月支出、订阅、项目管理、复盘全部压在同一行。

## 九、UI 确认原则

Capture 页的确认原则是：

1. AI 先做大致处理。
2. 用户只补红框字段，不需要重写整条。
3. `needs_review` 可以逐条允许提交。
4. `blocked` 条目单独修，不会拖住整批。

最轻量的补法：

- 缺开始/结束时间：只补时间。
- 缺时长：只补时长。
- 类别不对：只改类别。
- 这条不该入库：直接转复盘或忽略。

不需要为了一个字段，把整条原文重新整理一遍。

## 十、解析兜底原则

V4 的处理原则是提示词和规则双向约束：

1. AI 负责理解自然语言，抽取候选字段，不负责编造缺失字段。
2. Rust 规则层负责确定性兜底，包括 JSON 修复、时间锚点识别、金额/时长结构识别、必填检查、项目和标签绑定。
3. 单点时间默认是 `time_marker`，不算时间消耗。
4. 反思、计划、情绪、社交、健康状态默认是 `reference_note`，不强行入库。
5. 只有字段足够完整的草稿才进入可提交状态，其余都进入用户审核或引用保存。
