# 复盘素材编辑与历史补录模块化开发计划

## 1. 背景

当前复盘页可以展示 `review_notes`，AI 审核区也能在提交时创建复盘素材，但已入库素材缺少编辑接口。结果是用户在复盘页看到“复盘素材”后，无法修正标题、类型、正文或日期，也无法把误归类素材隐藏。

另一个问题是记录入口默认绑定今天。用户漏记后，需要能进入过去日期，补充当日时间、收入、支出记录；其中学习不再是独立记录类型，而是 `time_records.category_code = "learning"` 的时间记录。用户也需要能编辑过去日期已有记录，避免因为当天没录入导致数据永久断档。

本计划不做 MVP 式一次性串通，而是按模块拆分，每个模块可以独立开发、测试和回归。

## 1.1 前置事实：时间记录统一迁移后的口径

本计划基于已完成的时间记录统一迁移：

- 后续不再新增或恢复独立的 `learning_records` 写入路径。
- 后端、FFI、Flutter service 不再使用 `create_learning_record`、`update_learning_record`、`get_learning_record_snapshot`。
- 前端不再使用 `CaptureType.learning` 或 `RecordKind.learning`。
- 学习记录统一作为时间记录处理：
  - `category_code = "learning"`
  - `content` 表示学习内容
  - `application_level_code` 表示应用层级
- `review_notes.linked_record_kind` 只允许 `time`、`income`、`expense`；如果素材关联学习记录，也应关联到 `time`。
- AI 输入侧可以继续兼容 `learning` / `learning_record` 这些旧语义，但落库和编辑侧必须统一到 `TimeRecord`。

## 2. 当前代码状态

已存在能力：

- Rust 数据表：`review_notes` 已有 `id/user_id/occurred_on/note_type/title/content/source/visibility/confidence/raw_text/linked_record_kind/linked_record_id/created_at/updated_at`。
- Rust 后端：`ReviewNoteRepository::create/list_for_date/list_for_range` 和 `ReviewNoteService::create_note/list_notes_for_date/list_notes_for_range` 已存在。
- FFI：已有 `create_review_note`、`list_review_notes_for_date`、`list_review_notes_for_range`。
- 周期复盘：`ReviewRepository::build_report` 已经把 `ReviewNoteRepository::list_for_range` 合并进 `ReviewReport.review_notes`。
- 日详情页：`DayDetailPage` 已能按日期读取流水、编辑和删除三类记录：`time`、`income`、`expense`。学习通过 `time.category_code = "learning"` 表达。
- 记录更新：`update_time_record`、`update_income_record`、`update_expense_record` 已存在。

缺口：

- 缺 `update_review_note`。
- 缺 `delete_review_note` 或软删除入口。
- Flutter `RustApi/AppService/NativeRustApi` 没有复盘素材的 typed 方法。
- `DayDetailPage` 不加载也不编辑复盘素材。
- `DayDetailPage` 的“新增时间记录”只跳 `/capture`，没有把 `anchorDate` 传入录入页。
- `CapturePage` 默认使用 `runtime.todayDate`，缺明确的历史日期上下文入口。

## 3. 设计原则

- 复盘素材是独立领域对象，不混入 record 编辑器。
- 记录编辑只维护 `time`、`income`、`expense` 三类 record；学习相关编辑必须走时间记录编辑器，并保留 `category_code = "learning"`。
- 删除优先做软删除：将 `visibility` 更新为 `hidden`，保留原始素材和审计时间。
- 过去日期补录必须由明确的 `anchorDate` 驱动，不能靠运行时“今天”隐式推导。
- 所有编辑完成后，日详情、复盘报告、Today 相关聚合都通过重新查询刷新，不做前端本地猜测。
- 每一步都要有可验收边界：Rust 单测或集成测、Flutter 模型/服务测试、页面手动验证。

## 4. 模块 A：复盘素材后端编辑能力

目标：给已入库 `review_notes` 补齐更新和软删除接口。

数据模型：

- 新增 `UpdateReviewNoteInput`，字段建议复用可编辑字段：
  - `user_id`
  - `occurred_on`
  - `note_type`
  - `title`
  - `content`
  - `visibility`
  - `confidence`
  - `raw_text`
  - `linked_record_kind`
  - `linked_record_id`
- 可以先复用 `CreateReviewNoteInput` 作为 update payload 的 `input`，但建议单独建 `UpdateReviewNoteInput`，避免后续 create-only 字段和 update-only 字段混淆。

Repository：

- `ReviewNoteRepository::update(connection, note_id, input) -> Result<ReviewNote>`
  - 校验 `user_id` 存在。
  - 校验 note 属于该 user。
  - 校验 `linked_record_kind` 只能是 `time`、`income`、`expense`，不能重新引入 `learning`。
  - 更新 `occurred_on/note_type/title/content/visibility/confidence/raw_text/linked_record_kind/linked_record_id/updated_at`。
  - 返回更新后的完整 `ReviewNote`。
- `ReviewNoteRepository::hide(connection, user_id, note_id) -> Result<()>`
  - `visibility = 'hidden'`。
  - 更新 `updated_at`。

Service：

- `ReviewNoteService::update_note(note_id, input)`
- `ReviewNoteService::hide_note(user_id, note_id)`

FFI：

- `update_review_note`
  - payload: `{ "note_id": "...", "input": { ... } }`
  - response: `ReviewNote`
- `hide_review_note`
  - payload: `{ "user_id": "...", "note_id": "..." }`
  - response: `null` 或 `{ "ok": true }`

测试：

- 更新标题、正文、类型、日期后可按新日期查到，旧日期查不到。
- hide 后 `list_for_date/list_for_range/build_report` 都不返回该素材。
- 跨 user 更新或隐藏失败。
- 非法 `note_type/visibility/occurred_on` 失败。

## 5. 模块 B：Flutter 服务层与模型契约

目标：前端不通过散落的 `invokeRaw` 操作复盘素材，而是有清晰服务边界。

RustApi：

- `listReviewNotesForDate({ userId, occurredOn })`
- `createReviewNote({ input })`
- `updateReviewNote({ noteId, input })`
- `hideReviewNote({ userId, noteId })`

AppService：

- 透传以上能力。
- 统一 payload key 使用 Rust 约定：`occurred_on/note_type/raw_text/linked_record_kind/linked_record_id`。

NativeRustApi：

- 调用现有或新增 FFI method。
- 使用 `ReviewNoteModel.fromJson` 解析返回值。

模型：

- 为编辑表单增加 `ReviewNoteMutationInput` 或 `toMutationJson()`，避免页面手写 Map。
- 保留 `ReviewNoteModel` 的只读展示语义。

测试：

- `ReviewNoteModel.fromJson` 覆盖 nullable 字段。
- mutation payload key 与 Rust FFI 保持一致。

## 6. 模块 C：日详情页加入复盘素材区

目标：`/day/:date` 同时管理当日流水和复盘素材，成为“过去日期修正”的主入口。

加载：

- `DayDetailPage._load()` 同时读取：
  - `getRecordsForDate(date: anchorDate)`
  - `listReviewNotesForDate(occurredOn: anchorDate)`
- 页面状态从 `ViewState<List<RecentRecordItem>>` 升级为 `ViewState<DayDetailData>`：
  - `records`
  - `reviewNotes`

展示：

- 在“当日流水”下方增加“复盘素材”区。
- 空状态允许显示“暂无复盘素材”，但不能让整页进入不可用状态。
- 每条素材显示：
  - 类型 pill
  - title
  - content
  - source 或 linked record 提示
  - menu：编辑、隐藏

编辑：

- 新建 `ReviewNoteEditorDialog`，不要复用 `RecordEditorDialog`。
- 字段：
  - 发生日期
  - 类型
  - 标题
  - 内容
  - 可见性
  - 原始文本，可折叠或放低优先级
- 保存调用 `updateReviewNote`。
- 隐藏调用 `hideReviewNote`，确认弹窗文案用“隐藏素材”，避免误导为物理删除。

新增：

- 日详情页 action row 增加“新增复盘素材”。
- 默认 `occurred_on = widget.anchorDate`、`source = manual`、`visibility = normal`。
- 保存后刷新日详情。

导出：

- `buildDayDetailExportMetadata` 增加 `reviewNotes`，否则导出的当日详情会漏掉复盘素材。

## 7. 模块 D：过去日期补充记录

目标：从过去某天进入录入页时，所有新建记录都默认落到该日期。

路由与上下文：

- `/capture` 支持可选参数 `contextDate`。
- 从 `/day/:date` 点击“新增记录”时传入 `contextDate = anchorDate`。
- `CapturePage` 初始化时优先使用路由参数，其次才使用 `runtime.todayDate`。

手动录入：

- 时间、收入、支出表单默认日期全部使用 `contextDate`；学习补录通过时间表单选择 `category_code = "learning"` 完成。
- 时间段选择时，如果只选时间不选日期，日期必须从 `contextDate` 组装，而不是今天。
- 提交成功后，如果来源是日详情，返回或刷新 `/day/:contextDate`。

AI 解析：

- `parse_ai_input_v2` 和 `commit_ai_capture` 的 `context_date` 使用 `contextDate`。
- 审核区的复盘素材 `occurred_on` 也默认使用 `contextDate`。
- 页面显式显示当前录入日期，防止用户以为正在录今天。

快捷入口：

- Today 页仍默认今天。
- 日详情页负责过去日期补录。
- 后续可在记录管理页增加日期选择器，但不作为本模块阻塞项。

测试：

- 从 `/day/2026-04-20` 进入 capture，创建收入记录后 `occurred_on = 2026-04-20`。
- 创建 duration-only 时间记录时 `occurred_on = 2026-04-20`。
- 创建学习记录时实际提交 `create_time_record`，且 `category_code = "learning"`、`occurred_on = 2026-04-20`。
- 创建带 `started_at/ended_at` 的时间记录时，日期和时间组装不漂移。
- AI 解析提交复盘素材时 `occurred_on = 2026-04-20`。

## 8. 模块 E：日期浏览与补录体验

目标：让用户能自然找到过去日期并补录。

优先顺序：

1. 保持现有 `/day/:date` 能被直接路由访问。
2. 在记录管理页或日详情页增加日期选择能力。
3. 对空日期显示“新增记录”和“新增复盘素材”，即使 records 为空也能操作。

注意点：

- 当前 `DayDetailPage` 在 records 为空时会显示空态，后续要保证 action row 仍可用。
- 不要把“今天详情”和“日详情”拆成两套能力；Today 只做总览，日详情承担编辑和补录。

## 9. 模块 F：回归与验收

Rust：

- `cargo test review_note` 或对应集成测试。
- 覆盖 create/update/hide/list/report 全链路。

Flutter：

- `dart format` 涉及文件。
- 模型和 payload 测试。
- 页面手动验证：
  - 复盘页素材仍能展示。
  - 日详情能展示、编辑、隐藏素材。
  - 从过去日期补录时间、收入、支出记录后，日详情和复盘报告数据刷新；学习补录作为 `category_code = "learning"` 的时间记录验证。

数据一致性验收：

- 编辑素材日期后，素材从旧日期日详情移动到新日期日详情。
- 隐藏素材后，复盘页和日详情均不展示。
- 历史补录不会污染今天数据。
- 导出日详情包含 records 与 review notes。

## 10. 推荐开发顺序

1. 后端接口：`UpdateReviewNoteInput`、repository、service、FFI、Rust 测试。
2. Flutter 服务契约：`RustApi/AppService/NativeRustApi` typed 方法和 payload 模型。
3. 日详情数据结构：`DayDetailData` 合并 records 与 review notes。
4. 复盘素材 UI：列表、编辑弹窗、新增、隐藏。
5. 历史日期 capture 上下文：路由参数、默认日期、提交回跳。
6. 导出补齐：`buildDayDetailExportMetadata` 增加 review notes。
7. 回归：Rust tests、Flutter format/test、手动验证主要路径。

## 11. 暂不纳入本轮

- 复盘素材与具体 record 的强绑定编辑。
- 批量移动素材日期。
- 物理删除 review note。
- 复杂日历视图。
- 离线冲突合并策略。
