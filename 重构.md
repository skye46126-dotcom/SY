
# 1. 项目当前定位

SkyeOS 不是一个普通记账 App，也不是一个简单时间记录工具。

它更接近于一个：

```text
个人经营分析系统
```

它要解决的核心问题是：

```text
我把时间花在哪里？
我赚了多少钱？
我支出了多少钱？
我的项目值不值得继续？
我的被动收入能覆盖多少必要支出？
我的实际时薪和理想时薪差多少？
我的学习、工作、AI 辅助效率如何？
我在某个周期内的状态是否变好？
```

所以项目的核心主题不是“记录”，而是：

```text
用时间、金钱、项目、成本、效率数据，计算个人经营状态。
```

记录只是输入，分析和反馈才是核心价值。

## 2.2 AI 辅助录入

AI 辅助录入包含：

```text
自然语言解析输入
三引擎解析：规则引擎、LLM API、VCP 插件
AUTO 模式自动降级
解析结果草稿预览
用户确认后一键入库
AI 服务配置：Provider、Base URL、API Key、Model、Prompt
```

AI 辅助录入的核心原则：

```text
AI 只生成草稿，不直接入库。
```

正确流程：

```text
自然语言输入
↓
规则引擎 / LLM / VCP 插件解析
↓
生成 ParseDraft 草稿
↓
用户预览和修改
↓
用户确认
↓
UseCase / Rust Core 入库
```

AI 不能直接写数据库，否则会导致数据入口混乱。



```text

月回顾：本月第一天到本月最后一天
年回顾：1 月 1 日到 12 月 31 日
自定义回顾：用户选择的起止时间
```

```text

```

# 3. 为什么使用 Flutter + Rust 重构

## 3.1 当前问题

当前 Java 项目存在的问题：

```text
功能很多
代码写得很乱
部分页面逻辑在 Fragment 内部
Repository 中混合了 SQL 查询、业务计算、展示数据组织
UI、数据、分析、AI 逻辑边界不够清晰
后续维护和扩展困难
```

## 3.2 不建议直接在原 Java 项目里继续堆功能

如果继续在原 Java 项目上扩展，可能会出现：

```text
页面越来越重
Repository 越来越大
AI 解析、统计分析、项目 ROI 都混在一起
后期难以测试
后期难以迁移
```

## 3.3 Flutter + Rust 的定位

新的技术路线：

```text
Flutter 做 App 外壳和交互
Rust 做核心业务和数据计算
SQLite 做本地存储
旧 Java 项目作为功能参考和数据来源
```

一句话：

```text
Flutter 做脸，Rust 做脑子，SQLite 做记忆。
```

## 3.4 Flutter 适合做什么

Flutter 负责：

```text
页面 UI
导航
表单输入
列表展示
图表展示
用户确认
设置页面
状态管理
文件选择
权限申请
本地通知
用户交互相关逻辑
```

具体页面包括：

```text
Today 页面
Capture 页面
Management 页面
Review 页面
Project 页面
Settings 页面
AI 聊天页面
日详情页面
```

## 3.5 Rust 适合做什么

Rust 负责：

```text
数据模型
SQLite 读写
数据库迁移
记录创建和修改
标签绑定
项目关联
成本计算
今日总览统计
项目 ROI 分析
日/周/月/年回顾
自然语言解析后的结构校验
备份文件导出和恢复
指标计算
公式口径统一
```

Rust 不负责：

```text
页面 UI
按钮点击
列表渲染
动画
权限弹窗
Android 文件选择界面
```

# 4. 推荐最终架构

## 4.1 总体架构

```text
Flutter App
│
├── UI 层
│   页面、组件、导航、状态管理
│
├── Flutter 应用层
│   Riverpod / Provider / Bloc
│   调用 Rust 暴露的接口
│
├── Rust Core
│   记录系统
│   标签系统
│   项目系统
│   成本系统
│   指标系统
│   回顾系统
│   AI 草稿解析
│   备份恢复
│
├── SQLite 数据库
│   本地数据存储
│
└── 外部服务
    LLM API
    VCP Plugin
    FastAPI Backup Service
    云同步
```

## 4.2 分层职责

### Flutter UI 层

```text
展示数据
响应用户点击
收集表单输入
展示加载状态
展示错误状态
展示空状态
展示图表和列表
```

### Flutter Controller 层

```text
管理页面状态
调用 Rust API
处理 loading / error / success
把 Rust 返回 DTO 映射为页面状态
```

### Rust API / FFI 层

```text
暴露 Flutter 可调用接口
进行参数转换
返回结构化 DTO
隐藏内部数据库和计算细节
```

### Rust Service 层

```text
业务规则
指标计算
项目 ROI 计算
成本计算
回顾生成
AI 草稿校验
备份恢复
```

### Rust Repository 层

```text
SQLite 查询
SQLite 写入
事务处理
基础数据读取
不做复杂业务解释
```

### SQLite 层

```text
存储记录
存储标签
存储项目
存储设置
存储备份记录
存储迁移版本
```

```text

    └── BottomNavigationView
        ├── nav_today
```

主导航页有 4 个：

```text
nav_today -> TodayFragment
nav_capture -> CaptureFragment
nav_management -> ManagementFragment
nav_review -> ReviewFragment
```


打开。

返回键规则：

```text

如果当前在二级页面，优先 navigateUp()
如果 navigateUp() 失败，则回到 nav_management
```

# 13. 经营指标系统

## 13.1 重要性

这部分是项目最核心的链路。

它包括：

```text
时间
金钱
收入
支出
结构化成本
必要支出
项目成本
项目利润
项目 ROI
AI 辅助
效率
价值
状态
```

它不应该散落在页面、Repository 或 SQL 中。

它应该成为 Rust Core 中的核心领域模块。

推荐命名：

```text
OperatingMetrics
```

中文理解：

```text
个人经营指标系统
```

---

## 13.2 核心记号

```text
I = 收入总额，单位分
E = 直接支出总额，单位分
SE = 结构化支出总额，单位分
NE = 必要支出总额，单位分
PI = 被动收入总额，单位分
WM = 工作时长总分钟
TM = 总时长总分钟
LM = 学习时长总分钟
BHR = 基准时薪，单位分/小时
IHR = 理想时薪，单位分/小时
```

---

## 13.3 金额、时间、收益、支出公式

```text
总收入：I = Σ income.amount_cents
直接支出：E = Σ expense.amount_cents
总支出：TotalExpense = E + SE
工作时长：WM = Σ time_log.duration_minutes where category = work
总时长：TM = Σ time_log.duration_minutes
学习时长：LM = Σ learning_record.duration_minutes
实际时薪：ActualHourlyRate = I * 60 / WM
时间债：TimeDebt = IHR - ActualHourlyRate
被动收入覆盖率：PassiveCoverRatio = PI / NE
自由度金额：Freedom = PI - NE
收入变化率：IncomeChangeRatio = (CurrentIncome - PrevIncome) / PrevIncome
支出变化率：ExpenseChangeRatio = (CurrentExpense - PrevExpense) / PrevExpense
工作时长变化率：WorkChangeRatio = (CurrentWorkMinutes - PrevWorkMinutes) / PrevWorkMinutes
```

旧代码落点：

```text
core/data/src/main/java/com/example/skyeos/data/repository/SQLiteLifeOsMetricsRepository.kt:126
core/data/src/main/java/com/example/skyeos/data/repository/SQLiteLifeOsReviewRepository.java:184
```

---

## 13.4 成本、项目、ROI 公式

```text
月结构化成本：MonthlyStructuralCost = BasicLivingCost + RecurringCost + CapexMonthlyAmortized
窗口期结构化成本：SE = Σ (MonthlyStructuralCost * OverlapDays / MonthDays)
必要支出：NE = NecessaryExpenseRaw + StructuralNecessaryExpense
基准时薪优先级：BHR = LastYearHourlyRate > 0 ? LastYearHourlyRate : (IHR > 0 ? IHR : ActualHourlyRate)
项目时间成本：ProjectTimeCost = BHR * ProjectTimeMinutes / 60
项目分摊结构化成本：AllocatedStructuralCost = StructuralExpenseWindow * ProjectTimeMinutes / TotalWorkMinutes
项目运营成本：OperatingCost = ProjectExpense + ProjectTimeCost
项目全成本：FullyLoadedCost = OperatingCost + AllocatedStructuralCost
项目利润：Profit = ProjectIncome - FullyLoadedCost
盈亏平衡收入：BreakEvenIncome = FullyLoadedCost
项目时薪：ProjectHourlyRate = (ProjectIncome / 100) / (ProjectTimeMinutes / 60)
ROI 通用公式：ROI = (Income - Cost) / Cost * 100%
项目运营 ROI：OperatingROI = (ProjectIncome - OperatingCost) / OperatingCost * 100%
项目全成本 ROI：FullyLoadedROI = (ProjectIncome - FullyLoadedCost) / FullyLoadedCost * 100%
```

旧代码落点：

```text
core/data/src/main/java/com/example/skyeos/data/repository/SQLiteLifeOsReadRepository.java:357
core/data/src/main/java/com/example/skyeos/data/repository/SQLiteLifeOsReviewRepository.java:250
```

注意：

内部建议统一使用分和分钟。

项目时薪内部建议写成：

```text
ProjectHourlyRateCents = ProjectIncomeCents * 60 / ProjectTimeMinutes
```

Flutter 展示时再除以 100 转成元。

---

## 13.5 Capex、效率、价值、状态、AI 公式

```text
残值金额：ResidualCents = PurchaseAmount * ResidualRateBps / 10000
可摊销金额：AmortizableCents = PurchaseAmount - ResidualCents
月摊销额：MonthlyAmortizedCents = AmortizableCents / UsefulMonths
摊销起始月：StartMonth = YearMonth(PurchaseDate)
摊销结束月：EndMonth = StartMonth + UsefulMonths - 1
AI辅助分钟：AiAssistMinutes = Σ (duration_minutes * ai_assist_ratio / 100)
AI辅助率：AiAssistRate = AiAssistMinutes / TM
工作效率均值：WorkEfficiencyAvg = Σ(duration * efficiency_score) / Σ(duration)
学习效率均值：LearningEfficiencyAvg = Σ(duration * efficiency_score) / Σ(duration)
时间分配占比：CategoryPercentage = CategoryMinutes / TM * 100%
目标达成：GoalReached = workMinutes >= minWorkMinutes && learningMinutes >= minLearningMinutes
```

# 15. 公式实现原则

## 15.1 Repository 只取数，Service 负责计算

旧项目中，Repository 可能同时承担：

```text
SQL 查询
业务计算
展示数据组织
```

重构后应该改为：

```text
SQLite Repository：只负责查原始数据
Metrics Service：负责基础指标计算
Review Service：负责回顾报告
ProjectRoi Service：负责项目分析
Formula Engine：负责公式口径
```

正确链路：

```text
Flutter 页面
↓
Rust API: get_dashboard_metrics()
↓
MetricsService
↓
Repository 读取原始数据
↓
FormulaEngine 计算
↓
返回 DTO 给 Flutter 展示
```

## 15.2 公式集中放置

建议集中到：

```text
formulas.rs
```

示例：

```rust
pub fn actual_hourly_rate_cents(income_cents: i64, work_minutes: i64) -> Option<i64> {
    if work_minutes <= 0 {
        None
    } else {
        Some(income_cents * 60 / work_minutes)
    }
}

pub fn passive_cover_ratio(passive_income_cents: i64, necessary_expense_cents: i64) -> Option<f64> {
    if necessary_expense_cents <= 0 {
        None
    } else {
        Some(passive_income_cents as f64 / necessary_expense_cents as f64)
    }
}

pub fn roi(income_cents: i64, cost_cents: i64) -> Option<f64> {
    if cost_cents <= 0 {
        None
    } else {
        Some((income_cents - cost_cents) as f64 / cost_cents as f64)
    }
}
```

## 15.3 分母为 0 的处理

必须统一处理：

```text
WM = 0 时，ActualHourlyRate 不能直接算
NE = 0 时，PassiveCoverRatio 不能直接算
PrevIncome = 0 时，IncomeChangeRatio 不能直接算
Cost = 0 时，ROI 不能直接算
```

Rust 中建议使用：

```text
Option<T>
```

Flutter 展示时对应：

```text
暂无数据
无法计算
上期为 0，无法比较
```

不能错误显示为：

```text
0
0%
```

## 15.4 单位统一

内部统一：

```text
金额：cents，单位分
时间：minutes，单位分钟
比例：ratio 或 bps，不要混用
评分：0-10 或 0-100，必须统一
```

命名建议：

```text

income_cents
expense_cents
hourly_rate_cents
residual_rate_bps
ai_assist_percent
```

不要把 `ratio`、`percent`、`bps` 混在一起。

# 18. UI 优化建议

## 18.1 UI 的定位

UI 不只是好看。

UI 是用户交互的第一层，决定用户能不能持续使用。

SkyeOS 的 UI 应围绕这个主体验：

```text
快速记录 → 立即看到变化 → 周期复盘 → 调整行为
```

## 18.2 主导航建议

当前主导航：

```text
Today
Capture
Management
Review
```

建议保留。

用户心智：

```text
Today：今日经营状态
Capture：快速录入中心
Management：项目、成本、标签、设置
Review：周期复盘与趋势
```

可选中文命名：

```text
今日
记录
管理
复盘
```

底部导航不要超过 4 个。

## 18.3 Today 页面优化

Today 页面应该成为：

```text
个人经营仪表盘
```

Today 页面不要塞太多指标。

它的任务是让用户 5 秒内知道：

```text
今天有没有赚钱？
今天有没有浪费时间？
今天是否接近目标？
今天是否值得复盘？
```

推荐四层结构：

### 第一层：今日状态一句话

例如：

```text
今天净收入 ¥126，工作 3.5h，学习 1.2h，目标完成 72%。
```

或者：

```text
今日状态：收入不错，但工作时长不足。
```

### 第二层：核心 4 指标卡片

```text
净收入
工作时长
学习时长
自由度金额
```

更多指标放到详情：

```text
被动收入覆盖率
时间债
实际时薪
结构化成本
支出变化率
```

### 第三层：今日目标进度条

```text
工作目标：180 / 240 分钟
学习目标：60 / 90 分钟
```

使用进度条比纯文字更直观。

### 第四层：最近记录 + 快速补录

Recent Records 卡片保留。

每条记录建议提供：

```text
编辑
复制一条
删除
挂项目
加标签
```

原因：

```text
用户录错后如果不能快速修改，数据会越来越脏。
```

## 18.4 Capture 页面优化

Capture 是最重要的输入页面。

它的目标是减少录入压力。

### 建议 1：默认显示最常用 3 个

```text
常用：
时间
收入
支出

更多：
学习
工作
项目
```

也可以根据使用频率自动排序。

### 建议 2：最小表单 + 展开更多

收入记录第一屏只填：

```text
金额
来源
时间
```

高级字段折叠：

```text
类型
是否被动收入
标签
项目
备注
```

支出记录第一屏只填：

```text
金额
类别
备注
```

时间记录第一屏只填：

```text
开始时间
结束时间
类别
```

学习记录第一屏只填：

```text
内容
时长
```

高级字段折叠：

```text
应用等级
效率
AI 占比
标签
项目
备注
```

### 建议 3：底部固定保存按钮

移动端表单长时，保存按钮应固定在底部。

建议：

```text
保存记录
保存并继续
```

### 建议 4：AI Tab 改成先输入后确认

AI Tab 适合做成聊天式或卡片式。

流程：

```text
自然语言输入
↓
解析成草稿
↓
用户确认
↓
入库
```

示例草稿卡片：

```text
学习记录草稿
内容：电路
时长：120 分钟
效率：7
AI 占比：30%
项目：未绑定
标签：未绑定

[修改] [确认入库]
```

AI 不能让用户感觉“不知道它存了什么”。

## 18.5 Management 页面优化

当前 Management 有：

```text
Projects
Income
Expense
Time
Cost
Settings
```

建议改成分组管理：

```text
数据管理
- 收入流水
- 支出流水
- 时间记录

经营管理
- 项目管理
- 成本管理
- 标签管理

系统管理
- 设置
- 备份
- AI 配置
```

这样 Management 不会变成杂物箱。

## 18.6 Review 页面优化

Review 当前信息很多，需要渐进展示。

推荐三层结构：

### 第一层：复盘摘要

例如：

```text
本周总结：
收入比上周 +18%
支出比上周 -6%
工作时长下降 2.5h
项目 ClawBot ROI 最高
```

### 第二层：核心趋势

只展示最重要的：

```text
收入趋势
支出趋势
工作时长趋势
```

对应公式：

```text
IncomeChangeRatio
ExpenseChangeRatio
WorkChangeRatio
```

### 第三层：下钻分析

复杂模块放到详情：

```text
标签分析
项目 ROI
下沉项目
关键事件
历史流水
```

不要把所有分析一次铺开。

## 18.7 Project Detail 页面优化

项目详情页应强调：

```text
值不值得继续做
```

顶部建议增加项目判断卡：

```text
项目状态：值得继续
原因：ROI 为 42%，项目时薪高于基准时薪。
```

或者：

```text
项目状态：需要警惕
原因：投入时间 18h，但收入为 0，结构化成本持续上升。
```

项目页要回答：

```text
这个项目赚钱吗？
这个项目耗时间吗？
这个项目值得继续吗？
是否应该暂停？
```

## 18.8 Settings 页面优化

Settings 现在太重。

当前包含：

```text
语言
云服务
AI 服务
备份
状态
标签管理
```

Flutter 版应拆为：

```text
SettingsPage
├── 通用设置
├── AI 服务配置
├── 云同步配置
├── 备份与恢复
├── 标签管理
└── 数据库与迁移
```

Settings 首页只放入口，不放所有表单。

## 18.9 全局交互建议

### 金额展示

内部用分，UI 展示为：

```text
¥12.34
```

不要在用户界面显示：

```text
1234 分
```

### 比例展示

统一显示：

```text
被动收入覆盖率：72%
ROI：35.6%
AI 辅助率：28%
```

不要有的显示 0.72，有的显示 72%。

### 无法计算不要显示 0

例如分母为 0 时，不显示：

```text
实际时薪 ¥0
变化率 0%
```

应显示：

```text
暂无数据
无法计算
上期为 0，无法比较
```

### 复杂指标要加解释

例如：

```text
时间债
自由度金额
被动收入覆盖率
结构化成本
全成本 ROI
```

旁边加问号说明。

示例：

```text
时间债 = 理想时薪 - 实际时薪
表示当前收入效率距离理想状态还有多远。
```

### 保存成功反馈

不要只显示 Toast。

更好的反馈：

```text
已保存时间记录 90 分钟。
今日工作时长更新为 210 分钟。
```

### 空状态设计

项目页空状态：

```text
还没有项目。
创建一个项目后，你可以统计它的收入、支出、时间成本和 ROI。
[创建项目]
```

Review 页空状态：

```text
本周数据还不够。
至少记录 1 条收入、支出或时间记录后，可以生成复盘。
[去记录]
```

# 22. 不要犯的错误

## 22.1 不要直接全部重写

风险：

```text
旧项目还能用
新项目写了很久还不能用
最后两个都维护不了
```

正确做法：

```text
旧项目作为参考
新项目从最小闭环开始
逐步恢复基础功能
```

## 22.2 不要先做 AI

AI 很诱人，但不是地基。

先做 AI 会遇到：

```text
字段没定
标签没定
项目关联没定
记录类型没定
解析结果格式没定
```

## 22.3 不要先做复杂 UI 美化

页面漂亮不等于系统清晰。

优先级：

```text
数据层 > 业务层 > 指标层 > AI 层 > UI 美化
```

## 22.4 不要让 Flutter 写复杂计算

Flutter 不应该计算：

```text
ROI
结构化成本
时间债
自由度
被动收入覆盖率
项目全成本
趋势对比
```

这些都应该在 Rust Core 中计算。

## 22.5 不要让 Repository 做所有事情

Repository 不应该同时负责：

```text
SQL
业务规则
公式计算
页面 DTO 组织
```

应拆成：

```text
Repository：取数
Service：业务
Formula：公式
DTO：返回展示结构
```

# 23. 最终架构结论

最终路线：

```text
Flutter 做 App
Rust 做 Core

不要全量照搬旧代码
先做最小闭环
再逐步补功能
```

最先跑通的闭环：

```text
Flutter Capture 页面
↓
调用 Rust create_time_record
↓
Rust 写入 SQLite
↓
Flutter Today 页面
↓
调用 Rust get_today_overview
↓
展示今日总时长
```

项目核心模块：

```text
OperatingMetrics 个人经营指标系统
```

最重要的分工：

```text
UI 只负责显示和交互
AI 只生成草稿
UseCase / Service 决定业务
Repository 只管数据
Rust Core 负责稳定逻辑
SQLite 负责存储
```

最终目标：

```text
把 SkyeOS 从一个功能很多但代码混乱的 Java App，
重构成一个 Flutter + Rust 的个人经营分析系统。
```
