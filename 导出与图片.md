# SkyeOS 导出中心功能需求文档

## 一、功能背景

SkyeOS 的核心不是单纯记录数据，而是围绕用户的 **时间、收入、支出、学习、项目、标签、AI 协作、状态评分** 构建一个个人状态系统。

因此，SkyeOS 的导出功能不应该只是把数据库字段导出来，而应该分成三类：

```text
1. 数据备份导出
用于备份、迁移、恢复、AI 分析。

2. 可读报告导出
用于个人复盘、归档、阅读、发送给他人。

3. 状态海报导出
用于社交分享、博客展示、个人状态表达。
```

最终目标是让 SkyeOS 具备一个完整的 **Export Center 导出中心**，既能导出机器可读的数据，也能导出人类可读的报告，还能导出有美感、保护隐私、适合分享的状态海报。

# 五、数据备份导出详细需求

## 1. 目标

数据备份导出的核心是：

```text
完整
准确
可恢复
可迁移
可分析
```

它不是给普通用户直接阅读的，而是给系统、AI、Excel、Python 或未来恢复流程使用的。

---

## 2. 应导出的原始数据

优先导出这些核心表：

```text
time_records      时间记录
income_records    收入记录
expense_records   支出记录
learning_records  学习记录
projects          项目
tags              标签
record_tags       记录与标签关系
record_projects   记录与项目关系
settings          系统设置
```

如果当前数据库中还有其他核心表，也应纳入完整备份。

---

## 3. 应导出的统计摘要

除了原始数据，也建议导出 summary，方便用户打开后直接看到统计结果。

summary 可包含：

```text
总收入 I
直接支出 E
结构化支出 SE
必要支出 NE
被动收入 PI
工作时长 WM
总时长 TM
学习时长 LM
基准时薪 BHR
理想时薪 IHR
被动收入覆盖率
有效时薪
AI 协作占比
项目投入占比
学习投入占比
状态评分 state_score
价值评分 value_score
```

注意：

summary 是可重新计算的数据，但导出后可以提升可读性和分析便利性。

---

## 5. CSV 导出

CSV 用于：

```text
Excel 查看
WPS 查看
Python 分析
AI 读取

```

CSV 要求：

```text
每张主要数据表单独导出为一个 csv 文件
字段名尽量使用数据库字段名
时间字段使用 ISO 8601 格式
金额字段保留 amount_cents
可以额外提供 amount_display
关联关系表也要导出
```

CSV 文件示例：

```text

expense_records.csv
learning_records.csv
projects.csv

record_tags.csv
record_projects.csv

```

# 六、可读报告导出详细需求

## 1. 目标

可读报告不是原始数据导出，而是“人类可读的状态报告”。

它应该做到：

```text
有标题
有分区
有表格
有解释
有总结
有中文 / 英文说明
不是一串字符
不是 JSON 字段堆积
不是数据库字段原样展示
```

---

## 2. 支持格式

优先级：

```text
1. Markdown
2. TXT
3. PDF
```

建议优先实现：

```text
Markdown + TXT
```

PDF 可以后续由 Markdown 转换。

---

## 3. 支持语言

报告生成时提供语言选择：

```text
中文
English
中英双语
```

默认建议：

```text
中文
```

如果用于作品集、博客、求职展示，可以选择：

```text
中英双语
```

---

## 4. Markdown 文件命名

```text
skyeos-daily-report-2026-04-27.md
skyeos-weekly-report-2026-W18.md
skyeos-monthly-report-2026-04.md
```

如果有语言区分：


## 8. 报告内容模块

### Summary 总结

用于自然语言解释当前状态。

示例：

```text
今日整体状态稳定，主要精力投入在系统建设和个人效率提升上。
```

英文示例：

```text
Today was a focused and steady day. The main energy was spent on system building and personal productivity.
```

---

### Key Metrics 核心指标

展示最重要的状态数据。

包含：

```text
状态评分
专注时长
学习时长
工作时长
AI 协作占比
主投入方向
财务状态
```

---

### Time Review 时间回顾

解释时间如何分布。

可展示：

```text
专注任务时间
学习时间
工作时间
其他记录时间
项目投入时间
```

不建议公开默认展示：

```text
具体时间段
详细日程
原始备注
```

---

### Financial Review 财务回顾

财务数据非常敏感，默认不展示具体金额。

支持三种模式：

```text
隐藏金额
显示趋势
显示具体数值
```

隐藏金额示例：

```text
财务状态：Positive
支出控制：Good
结余状态：Stable
```

显示趋势示例：

```text
收入较上周 ↑ 18%
支出较上周 ↓ 7%
净状态：Positive
```

显示具体数值示例：

```text
收入：¥860
支出：¥320
净结余：¥540
```

---

### Project Review 项目回顾

项目名称可能敏感，支持三种显示方式：

```text
显示真实项目名
显示项目分类
匿名显示
```

示例：

```text
真实项目名：SkyeOS Export Module
项目分类：Personal System
匿名显示：Project A
```

---

### Tags / Keywords 标签关键词

不建议直接把所有原始标签暴露出来，而是可以转成更自然的关键词。

原始标签：

```text
skyeos / agent / income / low-energy / blog
```

可读关键词：

```text
Focus
Learning
System Building
AI Assisted
```

---

### Notes 备注

默认不导出原始备注。

可以提供选项：

```text
隐藏原始备注
导出摘要
导出完整备注
```

公开分享模式下，应该只展示摘要：

```text
今天整体稳定，有明确推进。
```

不应该展示原文：

```text
今天下午有点焦虑，但是完成了导出模块……
```

# 七、状态海报导出详细需求

## 1. 目标

状态海报不是截图，不是报表，而是一张适合分享的视觉海报。

它需要做到：

```text
有美感
有留白
有图片
有核心数据
有状态表达
有隐私保护
有 SkyeOS 品牌识别
```

---

## 2. 海报尺寸

推荐默认尺寸：

```text
1080 × 1350
```

适合：

```text
朋友圈
小红书
Instagram
博客卡片
移动端分享
```

可后续扩展：

```text
1080 × 1920 竖屏故事版
1200 × 628 横版分享图
1920 × 1080 桌面横版
```

---

## 3. 海报结构

海报由四层组成：

```text
1. 背景层
2. 主视觉层
3. 数据层
4. 品牌层
```

---

### 1. 背景层

用于决定整张图的情绪和视觉基调。

可以是：

```text
内置渐变背景
模糊照片背景
插画背景
项目主题背景
轻纹理背景
天空 / 桌面 / 城市 / 抽象科技图
```

作用：

```text
提供氛围
减少空洞感
让图片更像海报
```

---

### 2. 主视觉层

这是图片融合的重点。

可以使用：

```text
今日主项目封面图
关联画廊图片
代表当前状态的插画
成长 / 专注 / 学习 / 效率主题图
Agent 形象
Mascot 角色图
每日情绪图
状态图标
```

要求：

```text
主视觉不能喧宾夺主
不应堆叠太多图片
建议一张主图为主
最多一张辅助小图
```

---

### 3. 数据层

展示核心状态数据。

可包含：

```text
状态评分
专注时长
学习时长
工作时长
收入 / 支出
财务趋势
AI 协作占比
被动收入覆盖率
主项目
项目分类
关键词
一句自动总结
```

要求：

```text
不要像报表
不要堆太多字段
一张图只讲一个重点
控制在 4 到 6 个核心指标
```

---

### 4. 品牌层

加入 SkyeOS 识别。

可包含：

```text
SkyeOS
Daily / Weekly / Monthly
日期
Generated by SkyeOS
小 logo
小 icon
```

位置建议：

```text
左上角
右上角
底部角落
```

---

# 十、ExportPolicy 导出策略

## 1. 作用

建立统一的 ExportPolicy，控制所有导出内容是否显示敏感数据。

同一次导出中：

```text
Markdown 报告
TXT 报告
PNG 海报
ZIP 内部文件
```

都必须遵守同一份 ExportPolicy。

如果用户选择隐藏金额，那么报告和海报都不能出现具体金额。

如果用户选择匿名项目，那么报告和海报都不能出现真实项目名。

---

```json
{
  "privacy_mode": "public",
```

# 十六、完整开发需求描述

可以直接交给 AI 或开发者：

```text
为 SkyeOS 增加“导出中心 Export Center”功能，支持数据备份、可读报告和状态海报三种导出方式。

一、功能目标

SkyeOS 的导出功能需要分为三类：

1. 数据备份导出
用于数据备份、迁移、恢复和 AI 分析，格式包括 JSON、CSV、ZIP。

2. 可读报告导出
```

# 十七、最终理解

SkyeOS 的导出功能应该被设计成一个完整系统，而不是简单的“导出按钮”。

可以这样理解：

```text
JSON / CSV
机器可读，用于备份、迁移、恢复、分析。

Markdown / TXT
人类可读，用于复盘、归档、阅读。

PNG 海报
外部可见，用于分享、展示、表达状态。
```

其中最重要的设计原则是：

```text
数据导出要完整
报告导出要可读
海报导出要美观
分享导出要保护隐私
```

这样做之后，SkyeOS 的导出功能就不只是工具功能，而会成为整个系统的一个核心输出能力。

# 附录：当前已落地的页面图片导出实现

本文专门说明当前项目里“图片导出 / 图片文档”的目标、数据来源、技术分层和推进顺序。重点不是做一个临时截图按钮，而是把导出能力做成可以持续扩展的模块。

## 1. 目标

导出模块当前聚焦 3 件事：

1. 把当前页面的经营看板导出成高质量 PNG
2. 给每张图片同时产出一份结构化元数据文档
3. 让导出能力可以复用到 Today / Review / Project / Cost 等页面

当前不做的事：

- 不把图片导出塞进 Rust 或 SQLite
- 不先做 PDF 拼版
- 不先做云端分享链路

原因：

- 图片来自 Flutter 渲染结果，本质上是 UI 输出能力
- 业务数据已经通过 Rust 聚合完成，没必要再让 Rust 负责截图
- 先把页面级导出边界和文档格式定下来，后面扩展 PDF、分享、批量导出会更稳

## 2. 当前实现边界

已落地：

- `TodayPage`
  - 支持导出当前 dashboard 为 PNG
  - 同时导出同名 JSON 元数据
- `ReviewPage`
  - 支持导出当前 dashboard 为 PNG
  - 同时导出同名 JSON 元数据
- `ProjectDetailPage`
  - 支持导出项目经营详情图片文档
- `CostManagementPage`
  - 支持导出成本管理图片文档
- `DayDetailPage`
  - 支持导出单日流水明细图片文档
- `ExportCenterPage`
  - 汇总导出入口、默认导出目录、数据库备份入口
  - 展示最近导出的图片文档
  - 支持预览与删除
- `AppleDashboardPage(exportBoundaryKey)`
  - 统一提供页面导出边界
- `ModulePage(exportBoundaryKey)`
  - 让非 dashboard 页面也能接入同一套导出能力
- `ImageExportService`
  - 统一负责等待稳定帧、渲染、写文件、输出结果
  - 同时维护 `export_index.json`
- `export_metadata_builders.dart`
  - 统一输出 Today / Review / Project / Cost / Day Detail 的 JSON 元数据 schema

当前未落地：

- 多页面批量导出
- PDF / 长图分页 / 分享到系统

## 3. 数据流

图片导出不是一条新的数据库链，而是在既有数据链之上加一层 UI 输出：

`SQLite -> Rust repository/service -> FFI -> Flutter controller/page -> RepaintBoundary -> PNG + JSON`

拆开看：

1. 数据仍然由 Rust 提供
   - Today 使用 `TodayController.load`
   - Review 使用 `ReviewController.load`
   - Project / Cost 继续使用现有 service 查询链
2. Flutter 页面把数据渲染成 dashboard
3. `AppleDashboardPage` / `ModulePage` 用 `RepaintBoundary` 暴露统一的可导出区域
4. `ImageExportService` 把这个区域导出成：
   - `*.png`
   - `*.json`
5. `export_metadata_builders.dart` 生成结构化元数据
6. `ImageExportService` 更新本地 `export_index.json`
7. 页面弹出结果对话框，展示图片路径、元数据路径、导出目录
8. `ExportCenterPage` 从本地索引读取导出历史，用于回看与清理

## 4. 文件输出约定

输出内容：

- 一张 PNG 图片
- 一份 JSON 元数据文档

目录规则：

- 桌面端优先写入 `Downloads/SkyeOS/exports/<module>`
- 如果桌面下载目录不可用，则回退到应用文档目录
- 移动端直接写入应用文档目录下的 `SkyeOS/exports/<module>`

命名规则：

- `today-2026-04-27_20260427-153000.png`
- `today-2026-04-27_20260427-153000.json`

其中：

- 前半段是业务标题
- 后半段是导出时间戳

## 5. JSON 元数据内容

Today 导出元数据示例：

```json
{
  "page": "today",
  "anchor_date": "2026-04-27",
  "timezone": "Asia/Shanghai",
  "summary": {
    "cash_income_cents": 0,
    "cash_expense_cents": 0,
    "cash_balance_cents": 0,
    "total_time_minutes": 0,
    "total_work_minutes": 0,
    "total_learning_minutes": 0
  }
}
```

Review 导出元数据示例：

```json
{
  "page": "review",
  "window_kind": "week",
  "period_name": "2026-04-20 - 2026-04-26",
  "start_date": "2026-04-20",
  "end_date": "2026-04-26",
  "summary": {
    "total_income_cents": 0,
    "operating_expense_cents": 0,
    "operating_balance_cents": 0,
    "total_work_minutes": 0,
    "passive_cover_ratio": 0
  }
}
```

这样做的价值是：

- 图片用于展示和归档
- JSON 用于后续二次加工
  - 生成 PDF
  - 批量归档
  - 对接分享、同步或自动命名规则

## 6. UI 口径约束

导出出来的图片必须跟页面展示口径一致，所以页面文案同步做了约束：

- Today 使用现金流口径
  - `现金支出`
  - `现金结余`
- Review 使用经营口径
  - `经营支出`
  - `经营结余`

这一步很重要，因为如果页面上写“支出”，导出的图片会天然延续误解。

## 7. 模块化推进顺序

建议按下面顺序继续做，而不是一次性把所有导出都硬塞进去：

### 第 1 阶段：页面级 PNG + JSON

- Today
- Review
- Project Detail
- Cost Management
- 统一导出边界
- 统一落盘服务
- 统一元数据 builder
- 导出中心入口

### 第 2 阶段：扩到业务详情页

- Day Detail

这一阶段要处理的问题：

- 页面高度差异
- 明细列表过长
- 不同页面的元数据 schema

### 第 3 阶段：导出文档化

- 多张图片组成一个导出包
- 生成 `index.json`
- 生成归档清单
- 可选生成 Markdown / PDF

### 第 4 阶段：系统级流转

- 分享到系统
- 选择导出目录
- 云端同步导出物
- 定时导出

## 8. 现阶段取舍

当前实现有意做了这些取舍：

- 不新增数据库表
- 不走 Rust 导出接口
- 不先做 PDF
- 不先做图库权限适配

这是为了先把“导出对象是什么、页面边界在哪里、元数据怎么组织”这三件事定住。

## 9. 后续最值得做的事

1. 给 `ProjectDetailPage` 接入同一套导出服务
2. 把 `time_debt_cents` 在 UI 中继续收敛成“时薪差额”类命名
3. 为超长 Review 页面补分页或分段导出策略
4. 增加导出任务历史与重复导出去重规则

## 10. 一句话结论

导出模块现在的正确分层应该是：

`Rust 负责算数据，Flutter 负责画页面，导出服务负责把页面和数据说明一起落成文件。`
